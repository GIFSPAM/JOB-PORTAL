JOB PORTAL API STRUCTURE SUMMARY
================================

GOAL
----
Build a complete job portal API that supports:
1. Guest users viewing demo and public job listings
2. Jobseekers managing profiles, applications, resumes, skills, and saved jobs
3. Employers posting jobs and managing applicants
4. Admins verifying and auditing jobs

----------------------------------------------------------------
1. API GROUPS
----------------------------------------------------------------

PUBLIC APIS
-----------
Purpose:
- Expose job data that does not require login
- Power guest-facing public job browsing

Endpoints:
- GET /api/public/jobs
- GET /api/public/jobs/:job_id
- GET /api/public/skills
- GET /api/public/landing-jobs

Notes:
- /landing-jobs is recommended for guest demo job feeds
- public job responses should include company_name and skills

AUTH APIS
---------
Purpose:
- Register and authenticate users
- Return current logged-in user information

Endpoints:
- POST /api/auth/register
- POST /api/auth/login
- GET /api/auth/me

SEEKER APIS
-----------
Purpose:
- Support jobseeker-only features

Endpoints:
- POST /api/seeker/apply/:job_id
- GET /api/seeker/my-applications
- DELETE /api/seeker/revoke/:application_id
- GET /api/seeker/profile
- PUT /api/seeker/profile
- PUT /api/seeker/profile/resume
- PUT /api/seeker/skills
- POST /api/seeker/saved-jobs/:job_id
- GET /api/seeker/saved-jobs
- DELETE /api/seeker/saved-jobs/:job_id

EMPLOYER APIS
-------------
Purpose:
- Support job posting and applicant management

Endpoints:
- POST /api/employer/post
- GET /api/employer/my-jobs
- PUT /api/employer/update/:job_id
- PATCH /api/employer/status/:job_id
- DELETE /api/employer/delete-jobs/:job_id
- GET /api/employer/applicants/:job_id
- PATCH /api/employer/application-status/:application_id
- GET /api/employer/profile
- PUT /api/employer/profile

ADMIN APIS
----------
Purpose:
- Moderate jobs and manage platform users

Endpoints:
- GET /api/admin/users
- GET /api/admin/users/:user_id
- PATCH /api/admin/users/:user_id/status
- DELETE /api/admin/users/:user_id
- DELETE /api/admin/jobs/:job_id
- GET /api/admin/all-jobs
- PATCH /api/admin/verify-job/:job_id

FOLDER PURPOSES
---------------

config/
- Stores shared backend configuration
- db.js should manage the database connection pool

middleware/
- Stores reusable request logic before controllers run
- Example: token verification, role checks, upload handling

controllers/
- Stores grouped business logic by role or access type
- Each controller file can contain multiple handlers for that role
- Keeps logic centralized and easier to scan than many tiny controller files

routes/
- Stores separated route groups by role or access type
- Each role gets its own route file under its own route folder
- Keeps endpoint definitions clear without scattering controller logic

services/
- Optional layer for reusable role-based business logic or database operations
- Useful when controllers become large

shared/
- Stores helpers reused across multiple modules
- Keep only validators and response helpers for now

uploads/
- Stores generated uploaded files such as resumes
- Should contain files only, not source code

server.js
- Entry point of the backend
- Mounts middleware, grouped route files, static uploads, and the error handler

4. RESPONSIBILITY BY GROUPED CONTROLLERS
----------------------------------------------------------------

controllers/authControllers.js
- Handles registration, login, and current-user fetch
- Example endpoints:
	- POST /api/auth/register
	- POST /api/auth/login
	- GET /api/auth/me

controllers/publicControllers.js
- Handles guest-accessible APIs
- Example endpoints:
	- GET /api/public/jobs
	- GET /api/public/jobs/:job_id
	- GET /api/public/skills
	- GET /api/public/landing-jobs

controllers/seekerControllers.js
- Handles all jobseeker-only actions
- Suggested contents:
	- apply for job
	- list applications
	- revoke application
	- get and update profile
	- upload resume
	- update skills
	- save or unsave jobs

controllers/employerControllers.js
- Handles employer-only actions
- Suggested contents:
	- create job
	- list employer jobs
	- update job
	- update job status
	- delete job
	- list applicants
	- update application status
	- get and update company profile

controllers/adminControllers.js
- Handles moderation and admin-only management actions
- Suggested contents:
	- verify job
	- list all jobs
	- delete job as admin
	- list users
	- get single user
	- update user status
	- delete user

services/*.js
- Optional extracted business logic if controller files become too large
- Keep DB-heavy or reusable domain logic here

shared/validators/
- Shared request validation rules

shared/responses/
- Standard response format helpers if you want consistent JSON output
